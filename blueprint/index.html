PingoGate 开发蓝图

当前开发路线

先把 PingoGate 做成可运行、可替换、可观测的基础网关，再沿着平台治理和协议能力逐层扩展。

Roadmap-first

首个实现方向：网关地基

当前只确定先从最关键的热路径切入：SDK 兼容入口、Provider 原生透传、SSE、静态网关 Key、 YAML 配置、Nginx 风格 reload、结构化日志、Prometheus 指标和机器可读管理 API 都是候选范围。 管理面可以先用 bootstrap 静态鉴权，但需要预留未来 OAuth / OIDC 接入的身份边界。

OpenAI-compatible Anthropic Messages Gemini generateContent Rust Admin API
P1
先打通数据面和管理面最小闭环 单二进制、配置驱动、原生透传、基础鉴权、观测、热重载、Admin API 和身份抽象。
P2
再补控制面 SQLite、用户项目 Key、渠道模型管理、请求日志、控制台和第三方登录。
P3-P4
对齐主流网关能力 模型组、预算、fallback、成本核算、协议转换和上下文桥。
P5+
扩展到完整平台 文件、Batch、Realtime、Tools、Agentic APIs、多节点和企业身份。

兼容入口

公共路径保持 SDK 友好，优先实现 OpenAI-compatible、Claude/Gemini 原生入口。

运行时骨架

热路径依赖不可变 RuntimeSnapshot，reload 失败不影响当前流量。

治理能力

Key、项目、模型、渠道、限流、用量和审计是后续平台化候选能力，具体进入哪个阶段尚未决定。

协议桥接

上下文桥、Realtime、Batch 和工具生态先预留边界，后续按能力族落地。

终局形态

终局形态用于校准阶段取舍，确保早期地基不会堵住长期平台能力。

North Star

一句话目标

PingoGate 是统一 AI 流量入口：对客户端保持 SDK 友好，对平台提供治理能力， 对 Provider 保留完整 API 家族空间，对未来协议转换和上下文桥接留出核心架构位置。

1
替代现有 LLM Gateway 在 OpenAI-compatible、Claude、Gemini 等入口上达到 LiteLLM / New API 级覆盖。
2
统一模型平台治理 把用户、项目、Key、渠道、模型、额度、价格、用量、审计和策略放进同一个控制面。
3
跨协议与跨 Provider 桥接 让 OpenAI Responses、Gemini Interactions、Anthropic Messages 等不同语义能够被策略化转换。
4
保持轻量运维模型 默认单二进制、声明式配置、热重载；数据库、Redis、对象存储和多节点都是可选增强。

产品定位

PingoGate 同时承担网关、平台和协议转换层三种角色。

网关 + 平台 + 协议层

高性能 LLM 网关

基于 Rust 和 Pingora，处理鉴权、路由、限流、重试、回退、流式传输和上游调度。

多租户模型平台

长期支持用户、项目、API Key、额度、预算、价格、用量、审计和管理控制台。

多协议兼容层

兼容 OpenAI、Anthropic、Gemini 以及 OpenAI-compatible Provider，尽量让客户端只改 base URL 和 token。

差异化与边界

产品差异化来自高性能网关、低运维成本、治理能力和协议桥接空间。

Boundaries

比通用代理更强

不是只转发请求，而是治理 AI 流量。

比平台更轻

默认单二进制，不强制数据库。

比兼容层更稳

先保留协议语义，再做转换。

展开差异化细节

相对 LiteLLM / New API，PingoGate 的核心差异应落在 Rust + Pingora 热路径、单二进制部署、声明式 reload 和更清晰的 Provider / capability 边界。

相对普通 OpenAI-compatible proxy，PingoGate 不只追求接口兼容，还要承担项目、Key、预算、用量、审计、路由和故障治理。

相对重型平台，PingoGate 的默认形态必须轻：本地文件和内存 runtime 可启动，高级数据库、对象存储、多节点和企业身份作为增强能力进入。

设计原则

这些原则用于约束未来讨论，避免蓝图越做越大、越做越散。

长期约束

单二进制优先

默认产物是一个 pingogate，高级依赖只作为可选增强。

后端只用 Rust

网关、控制面和管理 API 都在 Rust 中实现。TypeScript 只用于嵌入式控制台前端。

SDK 兼容优先

公共路径应保持 provider-native 或 compatibility-native，不把 /openai/... 这类前缀作为主路径。

配置热重载优先

运行时扩展从 Nginx 风格的声明式 reload 开始，不把动态 native plugin 当作主路线。

请求路径管线化

Provider adapter 不拥有整个网关生命周期，只负责命名空间、能力和转换边界。

内容持久化受策略控制

默认记录元数据和指标。完整 prompt/response 存储必须由租户或项目显式开启。

AI 工作区和产品代码分离

仓库根目录只保留 AI 协作、规格、任务和自动化框架。网关、控制台、部署和产品文档进入独立产品工作区，避免 agent 上下文和实现代码互相污染。

不依赖 Rhai

主架构不引入 Rhai 作为默认脚本引擎。转换能力优先使用声明式 profile、受限 predicate 和 Rust 编译后的 TransformPlan；复杂扩展未来再考虑签名 WASM。

仓库与分发

PingoGate 是现代 AI coding 驱动的项目。根目录应作为 AI 协作壳，产品实现集中在独立工作区，并从同一源码生成不同分发物。

AI-first workspace

根目录职责

根目录只放 AI 框架和协作入口：AGENTS.md、Spec Kit、agent skills、工作流配置、顶层说明和许可。产品代码、产品文档、构建脚本和部署模板不直接散落在根目录。

产品工作区职责

非 AI 框架内容统一进入 product/，再按后端、前端、配置、部署、测试和产品文档分层。AI agent 通过规格和任务进入工作区，代码审查和发布从产品工作区产生。

开发仓库

主仓库保留 AI 规格、任务、蓝图和产品源码，适合 Codex / Spec Kit 持续迭代。根目录是协作层，product/ 是实现层。

源码分发

对外源码包可以裁剪 AI 协作目录，只保留 product/、许可、版本说明和必要构建入口，降低使用者理解成本。

运行分发

发布物来自同一产品工作区：单二进制、嵌入式控制台、Docker 镜像、Compose、Kubernetes/Helm 包和示例配置。

GitHub 分发

GitHub 是默认分发面：源码仓库承载 issue、PR、review 和 release；GitHub Releases 发布二进制、校验文件、配置样例和 changelog；容器镜像可同步发布到 GitHub Container Registry。

Marketplace / Catalog

Proxy 分发应像 Codex / Claude Code plugin marketplace：一个 catalog 文件列出可安装条目，每个条目指向独立 proxy 包、版本、来源、能力、安装策略和签名信息。

类 Docker Hub 模型

从用户视角，proxy marketplace 更像 Docker Hub：用户按命名空间、名称和 tag 拉取能力包，例如 official/deepseek:1.0 或 acme/internal-openai:2026-05，再在本地配置中启用。

官方仓库源

官方源由 product/proxies/catalog.json 暴露，随 release 发布并由 CI 校验。用户可直接使用官方源安装经过审查的 proxy、Provider profile、兼容路由和能力声明。

自建仓库源

企业或个人可以维护自己的 GitHub/GitLab 仓库源，暴露同样的 catalog 格式。PingoGate 通过 marketplace add / update 加载这些源，并允许按 ref、tag、sha 或签名策略固定版本。

贡献官方源

开发者通过 PR 向官方 product/proxies/ 增加内容。贡献必须包含 manifest、README、示例配置、兼容性测试、能力声明、维护者和安全边界，合并后进入官方 catalog。

分发仓库形态

可以保留一个 AI-first 主仓库，同时通过 GitHub Actions 生成面向使用者的 release artifact、官方 proxy catalog 和容器镜像。需要更干净的消费体验时，再派生只含产品源码和 catalog 的 distribution repository。

默认分发产物

官方 marketplace 起步分发 YAML/JSON profile 包，而不是 WASM。Profile 包描述 URL、headers、query、body、streaming、capabilities、示例配置和 conformance tests。

格式分工

人写配置使用 YAML：pingogate.yaml 和 proxy.yaml。机器索引和锁定使用 JSON：catalog.json、lock.json、JSON Schema。TOML 不作为核心 profile 格式。

Transform 管线

Proxy profile 应描述受限 HTTP/JSON/SSE 转换管线，显式支持 URL、header、query、body 的 set/remove/rename/default/map，而不是执行任意脚本。

架构总览

把复杂系统拆成稳定层次。每一层都有自己的职责，减少 Provider 适配器变成大泥球的风险。

分层架构

数据面 Data Plane

处理真实 AI 请求流量：协议识别、鉴权、路由、策略、转换、上游派发、流式响应、实时会话、观测和用量采集。

低延迟 高吞吐 热路径不可阻塞

控制面 Control Plane

管理平台状态：管理员维护租户、Provider、渠道、模型目录、价格和证书；用户或项目负责人自助管理项目 Key、预算、用量、请求日志和数据留存策略。

管理员治理 用户自助 权限边界

状态与仓库 State

分离能力仓库、运行配置和控制面状态。公开能力描述不存用户密钥，内容持久化必须受租户或项目策略控制。

PingoGate 不只按 endpoint 建模，而是按能力族建模。这样后续做路由、计费、策略和协议转换时更稳定。

Provider 范围

长期要保留完整 API 家族的架构空间。早期可以只实现少数 endpoint，但不能把系统设计成只会聊天补全。

三大 API 家族
Provider 近期落点 长期保留能力 设计提醒
OpenAI 优先 Chat Completions；Responses 仅做同 Provider 原生透传的候选项。 Responses、Conversations、Realtime、Audio、Images、Videos、Embeddings、Files、Vector Stores、Batches、Fine-tuning、Evals、Models、Moderations、Assistants、项目和组织管理。 不能把 OpenAI 适配器写死成 Chat Completions 专用。
Google Gemini 优先 generateContent；Interactions 仅做同 Provider 原生透传的候选项。 Interactions、Live、Batch、Embeddings、Files、cached content、token counting、图片、音频、视频、文档理解、Veo、函数调用、搜索 grounding、URL context、代码执行、OpenAI compatibility。 generateContent 是无状态生成入口，Interactions 和 Live 需要单独的状态与会话模型。
Anthropic 优先 Messages 和 SSE streaming。 Tool use、extended thinking、vision/document inputs、prompt caching、Message Batches、Files、Models、Usage/Admin、MCP connector、computer use、Claude Managed Agents、OpenAI SDK compatibility。 Messages 更接近全上下文无状态生成，不等同于 OpenAI Responses 的 previous_response_id 模型。

协议兼容与上下文桥

这是 PingoGate 的长期差异点：让不同 Provider 的协议语义可以被统一治理，而不是只做 URL 转发。

长期核心能力

三种 API 模式

  • 原生透传：客户端和上游使用同一 Provider 协议。
  • 兼容表面：客户端使用 OpenAI-compatible、Claude-compatible 或 Gemini-compatible 协议。
  • 协议桥：客户端协议和上游协议不同，需要转换请求、响应、流式事件、状态引用和用量。

推荐转换架构

Client Protocol Emulator -> Conversation Timeline -> Capability IR -> Upstream Materializer

不要为每一对协议单独堆一个庞大 converter。需要保留 Provider 特有逃生口，用于 reasoning trace、tool call、多模态、MCP 和 realtime 事件。

ClientStateRef

PingoGate 对客户端暴露的 response_id、interaction_id、session id 或自定义 conversation id。

ConversationTimeline

有序保存用户回合、工具调用、工具结果、助手输出、推理元数据引用、附件和 Provider 响应元数据。

ContextMaterializer

根据上游能力输出 messages、Anthropic blocks、Gemini contents、OpenAI input items、cache controls 或 Provider state ids。

为什么上下文桥重要
Codex 这类客户端可能使用 OpenAI Responses，并通过 previous_response_id 增量传递上下文。 但 DeepSeek 等上游可能只支持 Chat Completions，需要每次传完整 messages。 PingoGate 长期要能生成自己的 response id，保存必要的 conversation timeline，并在下一轮把上下文 materialize 成上游需要的完整请求。

平台能力

这些能力决定 PingoGate 能不能从代理变成平台。它们不会全部进入第一阶段，但需要在架构里预留位置。

从代理到平台

配置与热重载

  • 加载候选配置
  • schema 和语义校验
  • 解析密钥引用但不泄露密钥
  • 构建不可变 RuntimeSnapshot
  • 原子切换，失败不影响当前运行时

可观测与计费

  • 请求量、状态码、延迟、上游延迟
  • 输入、输出、reasoning、cache token
  • Provider 成本、租户价格、额度消耗
  • Prometheus、OpenTelemetry、结构化日志

租户与策略

  • Tenant、Project、User、Service Account
  • API Key、Policy、Routes、Channels
  • 预算、限流、并发、数据留存
  • 桥接、工具、实时、文件权限

管理员与用户管理

控制面需要同时支持企业管理员和普通用户。管理员负责全局 Provider、渠道、模型、价格、证书、reload 和审计；用户或项目负责人负责项目内 API Key、预算、用量、日志、文件和留存策略；普通用户可通过 BYOK 自带 Provider Key 和模型，完全私有。Phase 1 可以使用 bootstrap 静态管理员鉴权，但管理 API 必须通过内部 Principal / AuthContext / 授权动作边界，避免未来接入 OAuth、OIDC、GitHub、Google 或企业 SSO 时改写 API 语义。

文件与媒体

文件是基础设施，不是边缘功能。长期需要 PingoGateFileRef 统一抽象，映射租户、存储后端、Provider 文件引用、用途、类型、大小、留存和加密。

Batch Jobs

批处理需要 job 模型：创建、上传或引用输入、路由到 Provider、轮询或回调、归一化状态、记录用量和成本、按策略保留或清理结果。

嵌入式控制台

前端只做 TypeScript 静态资源，权限和校验留在 Rust 管理 API。控制台应区分管理员视图和用户视图：管理员处理全局配置，用户处理自己项目内的 Key、预算、用量、日志和文件。第三方登录属于长期控制台能力，前期只需要把登录结果映射到内部用户、角色、项目成员关系的模型边界留出来。

Decision log
Phase 1 主范围 尚未决定。当前只把 OpenAI Chat Completions、Anthropic Messages、Gemini generateContent 的原生透传视为首个实现方向的候选范围。
Responses / Interactions 是否进入首个阶段尚未决定；如果进入，倾向只考虑同 Provider 原生透传，不模拟状态或桥接到无状态上游。
配置格式 用户主配置和 proxy profile 使用 YAML；marketplace catalog、安装锁文件和 schema 使用 JSON。TOML 不作为核心 profile 格式。
管理 API 小型 Rust admin API 是首个实现方向的候选项，可能覆盖 health、readiness、config validation、reload 和 reload status；Phase 1 可使用 bootstrap 静态管理员鉴权，但必须经由内部 Principal / AuthContext / authorize(action, resource) 边界。
第三方登录 OAuth、OIDC、GitHub / Google 登录和企业 SSO 倾向延后到控制台和用户体系阶段；蓝图只要求早期管理 API 不把鉴权写死成全局 admin token 判断，未来登录结果应映射到内部用户、角色、项目成员关系和服务账号。
管理模型 控制面长期同时支持企业管理员和用户/项目负责人：管理员管理全局 Provider、渠道、模型、证书和 reload；用户管理自己项目内的 Key、预算、用量、日志和数据策略；普通用户可通过 BYOK 自带 Provider Key 和模型（完全私有）。
仓库分发 根目录作为 AI 协作壳，产品代码进入 product/；GitHub 承担源码协作、release 分发和 proxy/provider marketplace。用户可使用官方源、自建源，开发者可通过 PR 贡献官方 catalog。
Proxy 分发产物 默认分发 YAML/JSON 声明式 profile 包，描述 URL、header、query、body、streaming 和 capabilities。WASM 只作为未来高级扩展。
脚本引擎 放弃 Rhai 作为默认扩展路线。主路径使用声明式 TransformPlan 和 Rust 编译计划；复杂第三方逻辑未来优先考虑签名 WASM。
控制台时机 嵌入式 TypeScript 控制台暂未排期；倾向不进入首个实现阶段，首个阶段可只提供机器可读管理 API 和身份边界。
延后事项 跨协议转换、ConversationTimeline、realtime、batch、文件对象存储、billing、多租户数据库倾向延后，但最终阶段归属尚未决定，架构必须预留位置。
BYOK（Bring Your Own Key） 普通用户可在平台上放置自己的 Provider Key 和模型，完全私有，仅本人可用。通过 Channel Scope 分层实现：管理员管理 Platform Channel，用户自建 User Channel。BYOK 请求经过完整管线（用量统计、日志、审计、策略），但不走平台计费。Key 加密存储，管理员不可见明文。管理 API 和控制台同时提供。管理员可通过全局或租户级开关控制 BYOK 权限。
