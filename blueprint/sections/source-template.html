日期：2026-05-28

1. 终极目标 
## 1. 终极目标

PingoGate 的终极目标不是做一个简单的聊天代理，而是成为面向多模型时代的统一 AI 流量入口。

它应该让开发者继续使用熟悉的 SDK 和协议，同时让平台统一处理：

- 模型路由
- Provider 兼容
- 协议转换
- 上下文状态
- 文件和媒体
- 实时会话
- Batch 作业
- 工具和 Agent 能力
- 配额、预算、计费
- 日志、指标、审计
- 配置热重载和安全回滚

长期看，PingoGate 应该达到 LiteLLM / New API 级别的网关和平台能力覆盖，但实现方式要更偏向：

- Rust + Pingora 的高性能运行时
- 单二进制优先
- 声明式配置
- Nginx 风格热重载
- 清晰的 Provider 和能力边界
- 默认轻量、可选增强的部署模型

一句话：

> PingoGate 是一个 Rust + Pingora 驱动的 LLM Gateway / Proxy / Platform，用来统一 OpenAI、Gemini、Anthropic 和 OpenAI-compatible Provider 的请求入口、治理能力和未来协议桥接能力。
2. 产品定位 
## 2. 产品定位

PingoGate 同时承担三个角色。

### 2.1 高性能 LLM API 网关

负责处理真实流量路径上的网关能力：

- HTTP / HTTPS 入口
- SDK 兼容路径
- 鉴权
- 模型路由
- Provider 选择
- 限流
- 重试和回退
- 流式传输
- 上游健康检查
- 指标和日志

### 2.2 多租户 AI 模型平台

长期需要支持平台治理能力：

- 租户
- 项目
- 用户
- Service Account
- API Key
- Provider Key
- 模型目录
- 渠道管理
- 配额
- 预算
- 价格
- 用量
- 审计
- 管理控制台

### 2.3 多协议兼容和转换层

PingoGate 不应该只支持 OpenAI-compatible chat completion。

长期需要把这些 API 家族放进同一套架构里：

- OpenAI
- Google Gemini
- Anthropic
- OpenAI-compatible Provider
- 未来更多 Provider 和 Agent API

目标是让客户端尽量只改：

- base URL
- token
- 模型名或模型别名

而不是被迫重写业务调用路径。
2A. 产品规划视角 
## 2A. 产品规划视角

### 产品形态

- 开源单二进制网关：一个 `pingogate` 接管 AI 流量。
- 团队自托管控制台：集中管理项目、Key、Provider、模型、用量和日志。
- 企业 AI 流量治理平台：把预算、审计、身份和策略纳入治理。
- Provider / Proxy 能力仓库：用 profile、catalog 和未来签名能力包扩展模型生态。

### 核心用户

- 应用开发者：少改代码，接入统一网关。
- 团队管理员：集中管理 Key、模型、预算、用量和排障。
- 企业平台工程师：把 AI 流量纳入基础设施治理。
- AI Infra / DevOps：关注性能、观测、reload 和故障隔离。
- Provider / Proxy 贡献者：用 profile 和测试贡献 Provider 支持。
- 产品负责人：看清成本、趋势和能力覆盖。

### 核心工作流

- 接入 Provider：配置上游，验证后 reload。
- 创建项目和 Key：应用使用网关 Key，不直接接触 Provider Key。
- 替换客户端入口：只改 base URL、token 和模型名。
- 查看用量和日志：按项目、模型、Provider 和状态排查。
- 配置路由和 fallback：用策略降低上游波动影响。
- 管理身份和审计：从 bootstrap 管理员走向 OAuth / OIDC / SSO。

### 差异化与边界

- 比通用代理更强：不只转发请求，而是治理 AI 流量。
- 比重型平台更轻：默认单二进制，不强制数据库。
- 比单纯兼容层更稳：先保留协议语义，再做转换。
3. 设计原则 
## 3. 设计原则

### 3.1 单二进制优先

默认部署产物是一个 `pingogate` 二进制。

外部数据库、Redis、对象存储、Docker Compose、Kubernetes 等能力是增强项，不应该成为第一个可用版本的强制依赖。

### 3.2 后端只用 Rust

核心后端能力使用 Rust 实现。

TypeScript 只用于嵌入式 Web 控制台前端。所有权限、校验、策略和敏感操作都应留在 Rust 管理 API 中。

### 3.3 SDK 兼容优先

公共路径应保持 provider-native 或 compatibility-native。

不应把 `/openai/...`、`/anthropic/...`、`/google/...` 这类 Provider 前缀作为主 SDK 路径。

客户端应该尽量能沿用官方 SDK。

### 3.4 声明式配置和热重载优先

运行时扩展从 Nginx 风格的声明式 reload 开始，而不是把动态 native plugin 作为主路线。

核心 Provider 支持可以编译进二进制。未来如果要扩展插件能力，也应该先从能力仓库、WASM policy/converter、签名包等安全边界更清晰的方式开始。

### 3.5 请求路径管线化

Provider adapter 不应该拥有整个网关生命周期。

请求路径应拆成稳定管线：

```text
入口
-> 协议识别
-> 鉴权授权
-> 上下文解析
-> 路由
-> 策略
-> 请求转换
-> 上游派发
-> 响应转换
-> 上下文提交
-> 可观测性
-> 用量计费
```

Provider adapter 负责 Provider 命名空间、能力声明和 Provider 特有转换，不负责把整个系统绑死。

### 3.6 内容持久化受策略控制

默认只记录元数据、指标和必要审计信息。

完整 prompt / response 内容不应默认持久化。只有租户或项目策略明确允许时，才可以保存完整内容。

协议桥接需要历史上下文时，可以保存加密、短期、可过期的桥接状态。

### 3.7 外部依赖可选

默认开发和轻量生产模式可以使用 file + memory。

更高级部署可以选择：

- SQLite
- Postgres / MySQL
- Redis
- 对象存储
- Docker Compose
- Kubernetes

7. Nginx 风格热重载 
## 7. Nginx 风格热重载

PingoGate 应实现 Nginx 风格配置热重载，而不是把 native 动态库加载作为主路径。

### 7.1 触发方式

- `SIGHUP`
- admin API reload command
- file watcher
- repository revision update
- UI-triggered config publish

### 7.2 Reload 流程

1. 加载候选配置或仓库版本。
2. 解析并做 schema 校验。
3. 执行语义校验。
4. 解析密钥引用，但不暴露密钥值。
5. 校验证书、上游定义、模型路由、租户策略和 listener 变化。
6. 构建不可变 `RuntimeSnapshot`。
7. 执行 dry-run 检查和可选上游健康检查。
8. 原子切换 active snapshot。
9. 进行中的请求继续使用旧 snapshot。
10. 记录 reload revision、状态、错误和 rollback target。

### 7.3 实现方向

- 使用 `Arc` + 原子 snapshot holder，例如 `ArcSwap`。
- 新请求在接近入口处读取一次当前 snapshot。
- 旧请求持有旧 snapshot 直到完成。
- listener 变化由 listener manager 处理，因为端口和 socket 不能像路由表一样简单替换。
- reload 失败不能修改当前 active runtime。

### 7.4 热重载范围

- routes
- models
- channels
- upstreams
- auth policy
- static gateway keys
- rate limits
- budget policy
- metrics sinks
- Provider capability enablement
- certificate files
- domain mappings
- context bridge policy
- plugin / capability repository metadata

### 7.5 主路径不做的事

- 默认加载 Rust `.so`、`.dylib`、`.dll` 插件。
- 原地修改 live route table。
- 中断进行中的请求。

PingoGate 应按能力族建模 Provider，而不仅仅按 endpoint 建模。

Endpoint parity 很重要，但 capability parity 更重要。

- Chat Completions

- Realtime
- realtime sessions
- client secrets
- calls
- transcription sessions
- translations
- Audio speech / transcription / translation / voices / voice consent
- Images generation / edits / variations
- Videos 和 video content
- Embeddings
- Files 和 uploads
- Vector stores 和 vector store search
- Batches
- Fine-tuning jobs / checkpoints / events / permissions / pause / resume / cancel
- Evals 和 eval runs
- Containers 和 container files
- Models
- Moderations
- Skills
- Assistants / threads / messages / runs / run steps
- ChatKit sessions / threads
- Organization 和 project management APIs

早期不需要代理所有 OpenAI management API，但 OpenAI namespace 不应假设“只有生成接口”。

### 8.2 Google Gemini

长期适配器应预留这些 API 家族：

- `generateContent`
- streaming generation
- Interactions API
- Live API
- Batch API
- Embeddings
- Files API
- file input methods
- context caching / cached content

- Models
- text / image / audio / video / document understanding

- TTS through Gemini generation APIs
- function calling
- Google Search grounding
- URL context
- code execution
- file search
- built-in tools
- OpenAI compatibility surface

Gemini 同时有无状态和有状态表面：

- `generateContent` 是稳定无状态生成入口。
- Interactions 是有状态 agentic 入口。
- Live 是独立的长连接双向会话入口。

### 8.3 Anthropic

长期适配器应预留这些 API 家族：

10. 协议兼容和转换 
## 10. 协议兼容和转换

PingoGate 支持三种 API 模式。

### 10.1 原生透传 Native Pass-through

客户端使用 Provider 原生协议。

PingoGate 将请求转发到相同 Provider 协议，同时提供：

- 鉴权
- 路由
- 策略
- 指标
- 日志
- 审计

### 10.2 兼容表面 Compatibility Surface

客户端使用标准兼容协议，例如：

- OpenAI-compatible
- Claude-compatible
- Gemini-compatible

PingoGate 可以路由到原生支持该表面的 Provider，或路由到 converter。

### 10.3 协议桥 Protocol Bridge

客户端协议和上游协议不同。

PingoGate 需要转换：

- request
- response
- streaming events
- state references
- usage metadata

重要桥接路径：

- OpenAI Responses -> OpenAI Chat Completions
- OpenAI Responses -> Anthropic Messages
- OpenAI Responses -> Gemini generateContent
- Google Interactions -> Gemini generateContent
- Google Interactions -> OpenAI Chat Completions
- Google Interactions -> Anthropic Messages
- Chat Completions -> Responses
- generateContent -> Interactions
- Claude-compatible -> OpenAI-compatible / Gemini-compatible

推荐架构：

```text
Client Protocol Emulator
-> Conversation Timeline
-> Capability IR
-> Upstream Materializer
```

不要把每一对协议都做成一个独立的大 converter。

Provider 特有逃生口仍然必要，用于：

- reasoning traces
- tool calls
- multimodal inputs
- computer use
- MCP
- thinking signatures
- realtime session events
11. 上下文桥和会话状态虚拟化 
## 11. 上下文桥和会话状态虚拟化

上下文桥是 PingoGate 的长期核心差异点。

例子：

- Codex 使用 OpenAI Responses。
- 客户端通过 `previous_response_id` 增量传递上下文。
- DeepSeek 只支持 Chat Completions，需要每次传完整 `messages`。
- PingoGate 必须模拟 Responses API，生成自己的 response id，保存足够的 conversation timeline，并在下一轮 materialize 成完整 Chat Completions messages。

同类场景包括：

- Responses -> Anthropic Messages
- Responses -> Gemini generateContent
- Interactions -> Chat Completions
- Interactions -> Anthropic Messages
- Interactions -> generateContent

### 11.1 核心对象

`ClientStateRef`

- PingoGate 对客户端可见的 `response_id`
- `interaction_id`
- session id
- custom conversation id

`ConversationTimeline`

- 用户回合
- 工具调用
- 工具结果
- assistant outputs
- reasoning metadata references
- attachments
- provider response metadata

`ProviderStateRef`

- upstream response ids
- conversation ids
- interaction ids
- cached content ids
- prompt cache markers
- provider-specific cache metadata

`ContextMaterializer`

- 输出完整 messages
- Anthropic message blocks
- Gemini contents
- OpenAI input items
- cache controls
- provider state ids

`ContextPolicy`

- 内容持久化策略
- TTL
- 加密
- 脱敏
- 租户隔离
- 是否允许 bridge

### 11.2 默认策略

- 默认 metadata-first observability。
- 默认不保存完整 prompt / response 日志。
- 协议桥接需要历史内容时，可保存加密短期内容。
- 完整 conversation history management 是显式租户或项目功能。

如果租户完全禁用内容持久化，PingoGate 必须禁用需要历史 materialization 的桥接路径，只允许 native stateless / full-context 或 provider-stateful 路径。
12. Realtime 和 Live API 
## 12. Realtime 和 Live API

Realtime / Live API 需要独立于普通 request / response 和 SSE 的管线。

需要的概念：

- session lifecycle
- client secrets / ephemeral tokens
- WebSocket
- WebRTC
- bidirectional event stream
- audio frames
- text events
- video / image streams
- tool events
- session-scoped policy
- long-lived connection metrics
- backpressure
- idle timeout
- max session duration
- circuit breaking
- graceful close
- event translation

OpenAI Realtime 和 Gemini Live 应是一等能力族。

Anthropic 当前更适合通过 streaming Messages、tool workflows 和 agent workflows 接入，而不是映射成直接 Live API 等价物。

16A. BYOK（Bring Your Own Key） 
## 16A. BYOK（Bring Your Own Key）

管理员负责管理平台分发的模型和 Provider Key，但普通用户也应该能在平台上放置自己的模型和 Key。

### 16A.1 核心概念：Channel Scope

现有 Channel 概念扩展为分层模型：

```text
Channel
  ├── scope: platform     ← 管理员创建，全平台分发
  └── scope: user         ← 用户私有 BYOK
```

User Channel 和 Platform Channel 共享同一套数据结构和管线能力（Provider 类型、Key 加密、健康检查、重试、fallback、指标），区别只在所有权和可见性。

### 16A.2 数据模型

```text
UserProviderKey
  ├── id
  ├── owner_user_id        ← 所属用户
  ├── provider_type         ← openai / anthropic / gemini / openai-compatible
  ├── encrypted_key         ← 加密存储的 Provider API Key
  ├── base_url (optional)   ← 自定义 endpoint（适用 openai-compatible）
  ├── created_at
  └── last_used_at

UserChannel
  ├── id
  ├── owner_user_id
  ├── provider_key_id       ← 引用 UserProviderKey
  ├── name / alias
  ├── enabled_models[]      ← 用户声明此 Key 可用的模型列表
  ├── model_aliases{}       ← 可选的模型别名映射
  ├── priority / weight     ← 路由权重
  ├── status                ← active / disabled / error
  ├── health_check_policy   ← 可选健康检查策略
  └── policy_overrides      ← 用户级策略覆盖（受平台策略约束）
```

### 16A.3 路由层变化

请求管线中的模型路由步骤变为：

```text
请求进入 → 鉴权 → 识别用户身份
  → 查找 User Channel（当前用户 + 请求模型）
    → 有匹配 → 使用 User Channel 的 Provider Key 和上游配置
    → 无匹配 → 回落到 Platform Channel（正常平台路由）
```

管理员可以通过平台策略控制：

- 是否允许用户 BYOK（全局开关 / 租户级开关）
- 用户 BYOK 是否允许 fallback 到平台分发模型
- BYOK 请求是否受平台 rate limit / 配额约束

### 16A.4 隔离性

- 用户自带的 Key 和模型完全私有，只有本人可用。
- 不支持将 BYOK 渠道共享给其他用户或项目成员。
- 管理员无法查看用户的明文 Key（可见元信息：Provider 类型、创建时间、状态）。
- 管理员可以禁用特定用户或全局的 BYOK 能力。

### 16A.5 策略与治理

BYOK 请求仍然经过完整的请求管线：

- 用量统计：正常记录 token、延迟、状态码，但不走平台计费。
- 请求日志：正常记录。
- Rate Limit：受平台策略约束（可配置是否约束 BYOK）。
- 审计：正常审计。
- 模型白名单：用户 BYOK 声明的模型可受平台策略约束。

### 16A.6 安全边界

- 用户的 Provider Key 加密存储在平台数据库中，运行时解密使用。
- 用户只能 CRUD 自己的 UserProviderKey 和 UserChannel。
- BYOK 管理 API 必须通过 Principal / AuthContext / authorize(action, resource) 边界。

### 16A.7 控制台

用户控制台增加"我的模型"板块：

- 管理自己的 Provider Key（增删改，Key 加密存储，UI 只显示末尾 4 位）。
- 管理自己的 Channel（选择 Provider Key → 声明可用模型 → 启用/禁用）。
- 查看 BYOK 请求的用量和日志。
- 查看 BYOK Channel 的健康状态。

22. 通往终局的第一块地基 
## 22. 首个实现方向

首个实现方向的价值不是追平所有平台能力，而是验证正确骨架。具体交付范围仍需后续确认。

无论首个阶段怎么切，都应守住这些架构线：

- 公共路径保持 vendor-native 或 compatibility-native。
- 不把 `/openai/...`、`/anthropic/...`、`/google/...` 作为主 SDK 路径。
- 通过 path、headers 和 request shape 做协议识别。
- 按 protocol default + model/channel config 做路由。
- 热路径配置使用 immutable runtime snapshot。
- 即使 endpoint 少，也要保留 Provider namespace 边界。
- 指标标签至少覆盖 capability family、provider、upstream、model、tenant/project placeholder、status、streaming mode。
- 管理 API 即使只用 bootstrap 静态 token，也必须经过 Principal / AuthContext / authorize(action, resource) 抽象。
- 即使只实现 file/memory，也要设计 storage traits。
- 不支持的能力族要在配置和 API 响应中明确表达。
23. 当前已确认决策 
## 23. 当前已确认决策

- Phase 1 主范围尚未决定；OpenAI Chat Completions、Anthropic Messages、Gemini generateContent 的原生透传只是首个实现方向候选。
- Responses 和 Gemini Interactions 是否进入首个阶段尚未决定；如果进入，倾向只做同 Provider 原生透传。
- 首个阶段倾向不模拟 Responses / Interactions 状态。
- 首个阶段倾向不把 stateful API 桥接到 stateless upstream。
- 用户主配置和 proxy profile 使用 YAML；marketplace catalog、安装锁文件和 schema 使用 JSON。TOML 不作为核心 profile 格式。
- 小型 Rust admin API 是首个实现方向候选，可能覆盖 health、readiness、config validation、reload、reload status；Phase 1 可使用 bootstrap 静态管理员鉴权，但必须经由内部 Principal / AuthContext / authorize(action, resource) 边界。
- OAuth、OIDC、GitHub / Google 登录和企业 SSO 倾向延后到控制台和用户体系阶段；早期只预留外部身份映射到内部用户、角色、项目成员关系和服务账号的边界。
- 控制面长期同时支持企业管理员和用户 / 项目负责人；管理员管理全局资源，用户管理自己项目内的 Key、预算、用量、日志和数据策略。普通用户可通过 BYOK 自带 Provider Key 和模型，完全私有，仅本人可用。
- 根目录作为 AI 协作壳，产品代码进入 `product/`；GitHub 承担源码协作、release 分发和 proxy/provider marketplace。
- 默认分发 YAML/JSON 声明式 profile 包，描述 URL、header、query、body、streaming 和 capabilities。WASM 只作为未来高级扩展。
- 放弃 Rhai 作为默认扩展路线。主路径使用声明式 TransformPlan 和 Rust 编译计划；复杂第三方逻辑未来优先考虑签名 WASM。
- 嵌入式 TypeScript 控制台暂未排期，倾向不进入首个实现阶段。
- 首个实现阶段可只暴露机器可读 admin APIs 和身份边界，不要求 UI。
- 跨协议转换、ConversationTimeline、realtime/live、batch、files/object store、billing、多租户数据库倾向延后，但最终阶段归属尚未决定，架构中必须预留位置。
- BYOK（Bring Your Own Key）：普通用户可在平台上放置自己的 Provider Key 和模型，完全私有。通过 Channel Scope 分层实现（scope: platform / user），复用现有 Channel 管线能力。BYOK 请求经过完整管线但不走平台计费。Key 加密存储，管理员不可见明文。管理 API 和控制台同时提供 BYOK 管理入口。
