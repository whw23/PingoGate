Provider 扩展与插件架构讨论记录

本次讨论围绕 PingoGate 如何支持多 Provider 扩展展开：普通贡献者是否只需写 schema/transform、 插件应该以何种形态分发、转换模型如何设计、以及认证等边界问题。

Discussion

日期：2026-06-29

状态：讨论稿，尚未形成最终 spec

核心结论：倾向 M² 双向 adapter 模型，复杂转换通过预编译插件（WASM 优先）分发，简单 OpenAI-compatible 透传/改写由 Core 直接处理。

项目背景

PingoGate 当前处于蓝图阶段，尚未进入具体实现。

Context

目标

基于 Rust + Pingora 的高性能 LLM Gateway / Proxy，作为 LiteLLM / New API 的替代品，统一 OpenAI、Gemini、Anthropic 等 Provider 的 AI 流量入口。

当前状态

仓库仅包含 HTML 蓝图与文档，无后端 Rust 代码。已有 Spec Kit 工作流（.specify/、.claude/skills/）可用于后续规格化。

Provider 扩展愿景

让普通贡献者无需深入 Rust，也能为 PingoGate 增加新的 Provider 支持。

Vision

核心主张：

  • 贡献者只写 schema 和 transform。
  • 通过工具链或 CI/CD 把这些描述物打包成插件。
  • 用户通过命令安装插件，无需本地 Rust 工具链。
  • 运行期只加载插件，不解释 transform 源码。

插件形态对比

围绕性能、安全、贡献门槛和分发方式，比较了几种候选方案。

Options

内置 Rust crate

由核心团队维护，编译进主二进制。性能最高，但贡献门槛高、发版耦合。

动态库 .so/.dylib/.dll

接近原生性能，可热加载，但跨平台分发麻烦、无沙箱、ABI 版本管理成本高。

WASM 预编译插件

跨平台、有沙箱、性能接近原生。需要 WASM runtime（推荐 wasmtime），工具链较复杂。

声明式模板引擎

零工具链，适合简单字段映射，但无法表达深层重组、I/O、流式事件和状态管理。

方案 速度 安全 扩展性 贡献门槛
内置 Rust⭐⭐⭐⭐⭐⭐高
动态库⭐⭐⭐⭐⭐⭐⭐⭐高
WASM⭐⭐⭐⭐⭐⭐⭐⭐⭐中
声明式模板⭐⭐⭐⭐⭐⭐⭐最低
展开：为什么网关场景不推荐运行时解释 transform

网关数据面要求低延迟、高吞吐。运行时解释 transform（如 Rhai、Lua）意味着每次请求都要解释执行 AST/字节码，会比编译执行慢 5~10 倍。

WASM 的优势在于可以提前编译成机器码，运行时只需边界调用和内存拷贝，延迟可控在亚毫秒级。

转换模型：点对点 vs Canonical

讨论了三种转换组织方式。

Model

Canonical 中间格式

所有协议先转成内部标准格式，再转成目标格式。复杂度 2N，但跨 family 请求会 ×2 转换，且 Canonical 会随能力增长而膨胀。

M² 点对点

每个 adapter 直接表达 A → B。单次转换、语义精确，但 N 个协议最多产生 N² 种组合。

双向 adapter

一个模块同时处理 A ↔ B。第一阶段客户端入口固定为 OpenAI-compatible，每个上游 Provider 只需一个 openai ↔ provider 双向 adapter。

讨论结论：倾向 M² 双向 adapter 模型。第一阶段只支持 OpenAI-compatible 客户端入口，每个 Provider 贡献一个双向 adapter；未来若开放 Anthropic/Gemini 入口，再按需补齐组合。

透传 vs 转换路径

不是所有请求都需要转换，应分层处理。

Path 
Client Request
    │
    ▼
Protocol Parse
    │
    ▼
Route Match (model alias → provider)
    │
    ▼
Provider Profile Resolution
    │
    ├─ OpenAI-compatible, no body transform ──► 透传（最快路径）
    │
    ├─ OpenAI-compatible, simple rewrite ──────► L1 配置层改写
    │
    └─ Cross-family or complex semantics ──────► 转换路径（插件）

透传

只改 URL 和 auth header，body 直接流式转发。不经过任何 transform，性能最高。

L1 配置改写

改 model 别名、endpoint、header、query。由 Core 直接执行，无需插件。

插件转换

处理跨 family 的 body/response 结构重组、流式事件、错误包装等复杂语义。

插件分发模式

用户不能装 Rust，因此编译必须在贡献者侧或 CI 侧完成。

Distribution

推荐：WASM 预编译分发

贡献者仓库通过 CI 生成 .wasm，用户执行 pingogate plugin add <url> 即可下载加载。跨平台、有沙箱、用户零工具链。

备选：动态库预编译分发

性能接近原生，但需要按平台分发（linux-x86_64、linux-aarch64、darwin-arm64 等），且无沙箱。

Registry 设想：

  • 集中索引分散的 Provider 仓库。
  • 校验 schema/transform、签名、哈希。
  • 提供官方源和自建源两种模式。
  • 企业可配置白名单，只安装可信来源插件。

Provider 认证设计

认证由 Core 统一执行，不应写在 adapter 里。

Auth
Provider 认证方式
OpenAI / DeepSeek / 多数兼容Authorization: Bearer <token>
Azure OpenAIapi-key: <key> 或 Bearer
Anthropicx-api-key: <key> + anthropic-version
GeminiURL query ?key=<key>
AWS BedrockAWS SigV4 签名
GCP Vertex AIOAuth 2.0 / service account

设计原则：

  • Provider profile 声明认证类型和模板。
  • Core 在发请求前注入认证，adapter 不接触 Provider Key。
  • Gateway Key（用户访问 PingoGate）与 Provider Key（PingoGate 访问上游）严格分离。
  • Provider Key 优先从环境变量读取，生产环境建议对接密钥管理服务。

当前倾向架构

综合讨论后的暂定方向，待进一步确认。

Direction

Core 职责

HTTP 入口、鉴权、路由、限流、重试、日志、Provider Key 管理、L1 配置改写、透传路径。
